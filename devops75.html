The Infrastructure Playbook

A practical guide to the nine things you must set up on AWS before running any workload. These aren't optional best practices — they are the foundation. Skip them and you'll pay for it later, either in money, security incidents, or a 2am outage.

01
AWS Organizations — Multi-Account Structure
The first decision that makes or breaks your compliance.
AWS

The moment you sign up for AWS, you have a management account (also called the root account). Most people make the mistake of running everything in this one account — their application, their databases, their CI/CD pipelines, all of it. This is a problem that compounds over time.

Why a single account is a liability

When everything runs in one account, a misconfigured IAM role can give a developer access to production. A runaway process in dev can rack up costs that are indistinguishable from prod spend. There's no blast radius — a mistake anywhere affects everything. Compliance frameworks like SOC 2 and ISO 27001 explicitly require environment separation, and you cannot pass an audit with a single-account setup.

How to structure it

Management AccountBilling, SCPs, and org-level controls only. No workloads. No EC2. No RDS. Nothing.
Common Services AccountInternal tooling — Jenkins, SonarQube, Nexus, Artifactory, Vault. Shared infrastructure that every environment needs access to.
Environment AccountsOne per environment: Dev, QA, UAT, Staging, Prod, DR. Your actual workloads live here, isolated from each other.

Networking between accounts

The Common Services account should be VPC-peered with each environment account so your pipelines, artifact repositories, and internal tools can reach them. But here is a critical rule: environment accounts must not be peered with each other. Dev should never have a network route to Prod. This is both a security requirement and a compliance one — lateral movement between environments is a finding in every serious audit.

⚠️ Common mistake: Creating the account structure after you've already deployed workloads. Migrating resources between AWS accounts is painful and time-consuming. Get this right on day one — retrofitting an org structure onto a running system is months of work.

02
IAM Identity Center
Stop creating IAM users. There's a better way and it takes 10 minutes.
Security

The classic approach most tutorials teach is: go to IAM, create a user, attach a policy, hand over an access key. This works. But it creates a mess that grows quietly in the background until it bites you. Access keys get shared over Slack. Users leave and their credentials sit active for months. Nobody remembers which policies are attached to which users. MFA is optional and gets skipped.

What IAM Identity Center does differently

IAM Identity Center is the centralised access management layer for your entire AWS Organisation. Instead of creating IAM users inside each account, you manage all your people in one place and assign them permission sets — predefined roles — across whichever member accounts they need. Access is tied to an email identity, not a static credential.

How to set it up

01Enable IAM Identity Center from the management account. It's a one-click enable in the AWS console under Security, Identity & Compliance.
02Choose your identity source. If you use Google Workspace, Okta, or Azure AD, connect it via SAML so users log in with their existing credentials. If you have no IdP, use the built-in Identity Center directory.
03Create Permission Sets — these are the equivalent of roles. Common ones: ReadOnly, Developer, DevOps, Billing, Admin. Scope each one tightly using AWS managed policies or custom inline policies.
04Assign users or groups to accounts with the appropriate permission set. A developer gets Developer access to Dev and QA. They do not get access to Prod unless explicitly assigned.
05Share the AWS access portal URL with your team. They log in with their email, see only the accounts they have access to, and can assume their role via browser or CLI using aws sso login.

What this solves automatically

MFA is enforced by default — users cannot log in without it. When someone leaves, you disable or remove them from your IdP and their access to every AWS account is gone instantly. No access key rotation. No trawling through five accounts to remove an IAM user. No forgotten credentials.

💡 If you already have IAM users: Don't delete them on day one. Audit who is using them, migrate those users to Identity Center, verify access is working, then delete. Rushing this can lock people out of production.

03
Lock the Root Account
PEAK SECURITY PRIORITY. NO EXCEPTIONS.
Critical

The root account is the AWS account you signed up with using an email and password. It has absolute, unrestricted access to everything in your AWS Organisation. No SCP can restrict it. No permission boundary can limit it. No IAM policy applies to it. It is above every control mechanism you will ever put in place.

What root access can do that nothing else can

Delete member accountsPermanently. With all their resources. There is no undo.
Remove billing controlsDetach consolidated billing, remove budget alerts, disable Cost Explorer.
Disable CloudTrail org-wideWipe the audit trail retroactively and stop all future logging.
Close the AWS accountOnly root can close an account. Once closed, recovery is extremely limited.

How to lock it down

01Delete all root access keys immediately. Go to the root account's Security Credentials page and remove any existing access keys. Root should never authenticate via API — only via the console in emergencies.
02Enable MFA on the root account using a hardware security key (YubiKey or similar) if possible. A TOTP authenticator app is acceptable. SMS is not acceptable — SIM-swapping is a real attack vector.
03Store root credentials in a password manager with access restricted to two or three named individuals maximum. Document who has access. Treat it like physical vault access.
04Create a CloudWatch alarm that triggers when root account login activity is detected in CloudTrail. Any root login — even a legitimate one — should generate an immediate alert to your security team.
05Use a dedicated email alias for the root account that forwards to a distribution list, not a personal inbox. If the owner of that personal email leaves, you risk losing access to the account entirely.
🔴 Treat root as a break-glass account only. It should be used a handful of times per year at most — for tasks that explicitly require root such as changing account settings, enabling certain AWS services, or account recovery. Every use should be logged and reviewed.

04
Cost Anomaly Detection & Budget Alerts
Get notified before your costs explode out of control.
Cost

AWS bills are settled at the end of the month. Without active cost monitoring, a single misconfigured resource — a NAT gateway left running, a data transfer charge from an accidentally public bucket, a forgotten GPU instance — can generate a bill that arrives four weeks after the mistake was made. By then, the damage is done.

Cost Anomaly Detection

This is an ML-based service that builds a baseline of your normal spend patterns per service, per account, per region, and per cost allocation tag. When it detects a spend pattern that deviates significantly from that baseline, it alerts you — often within hours of the spike starting, not at the end of the month.

01Enable it from the management account via AWS Cost Management → Cost Anomaly Detection. Create monitors at the org level so it tracks all member accounts.
02Create alert subscriptions with an absolute or percentage threshold — for example, alert me when anomaly impact exceeds $50 in any 24-hour period. Route alerts to an SNS topic connected to email or Slack.
03Expect false positives for the first 4–8 weeks. The model is calibrating. Do not turn it off. As it learns your baseline, alerts become significantly more accurate and actionable.

Monthly Budget Alerts

Anomaly Detection is reactive — it catches spikes as they happen. Budget alerts are proactive — they tell you when you're trending toward a threshold. Set them up per account, not just at the org level. A dev account burning $800 in a month is invisible in org-level aggregates but very visible at the account level.

01Go to AWS Budgets in the management account. Create a budget per member account using a cost budget type.
02Set threshold alerts at 50%, 80%, and 100% of your expected monthly spend. At 50% you're tracking normally. At 80% you investigate. At 100% you act.
03Also set a forecasted spend alert — this triggers when AWS forecasts that you will exceed budget by end of month, even if you haven't crossed it yet.
ℹ️ Tag everything from day one. Cost Anomaly Detection becomes dramatically more useful when you can attribute spikes to a specific team, service, or environment via cost allocation tags. An alert that says "EC2 spend increased by $400" is far less useful than "EC2 spend for the payments service in prod increased by $400."

GuardDuty
Someone is mining crypto on your AWS bill right now. Would you even know?
Security

GuardDuty is AWS's threat detection service. It runs continuously in the background, ingesting data from CloudTrail management events, VPC Flow Logs, and DNS resolver logs, and uses machine learning to identify patterns that indicate a security threat. It does not require you to configure log ingestion yourself — enabling it is enough.

What it detects

CryptominingEC2 instances or Lambda functions communicating with known cryptocurrency mining pools. One of the most common attacks on compromised AWS accounts.
Credential exfiltrationAPI calls being made from unusual geographic locations or Tor exit nodes using your IAM credentials, suggesting they've been stolen.
ReconnaissanceUnusual port scanning or probing of your VPC resources from external IPs.
Anomalous IAM activityAn IAM role suddenly making API calls it has never made before — creating users, modifying security groups, disabling logging.
Instance compromiseEC2 instances communicating with known malicious IPs or command-and-control domains.

Setting up alerting properly

01Enable GuardDuty from the management account and delegate it to a security account (or the management account itself). This enables it across all member accounts automatically.
02Create an EventBridge rule that matches GuardDuty findings with severity HIGH or CRITICAL. Route these to an SNS topic. Wire SNS to email, PagerDuty, or a Slack channel via Lambda.
03For MEDIUM severity findings, consider routing to a daily digest rather than immediate alerts. Medium findings are worth reviewing but rarely require waking someone up at 3am.
⚠️ GuardDuty does not block anything. It detects and alerts. The response — isolating an instance, revoking credentials, blocking an IP — is yours to execute. Consider using GuardDuty findings to trigger automated Lambda-based remediation for the highest-severity finding types.

Inspector
Your infra is being scanned by attackers daily. Are you scanning it first?
Security

GuardDuty watches for active threats. Inspector looks for the vulnerabilities that would make those threats possible in the first place. It continuously scans your EC2 instances, Lambda functions, and container images stored in ECR for known CVEs, software vulnerabilities, and unintended network reachability.

What Inspector covers

EC2 instancesOS-level vulnerabilities in your running instances — unpatched kernel versions, vulnerable packages, exposed ports that shouldn't be reachable from the internet.
Container images (ECR)Every image pushed to ECR is automatically scanned against the NVD (National Vulnerability Database). You get a finding for every CVE present in your image layers.
Lambda functionsPackage dependencies in your Lambda deployment packages are scanned for known vulnerabilities — the same CVEs that affect your Docker images can exist in your serverless functions.

The key behaviour to understand

Inspector does not require you to initiate scans. Once enabled, it rescans automatically whenever the NVD is updated. This means a package that was clean when you deployed it last month can generate a new finding today because a CVE was published overnight. You don't need to do anything — you'll just receive the finding.

01Enable Inspector from the management account to cover all member accounts. Go to Amazon Inspector → Get Started and enable it org-wide.
02Route CRITICAL and HIGH severity findings to your alerting pipeline via EventBridge, the same way you route GuardDuty findings.
03Integrate Inspector findings into your CI/CD pipeline. Before promoting an image from QA to Prod, check if Inspector has raised any critical findings on that image. Fail the pipeline if it has.
💡 Don't try to fix everything at once. A fresh Inspector enable on a real workload will surface dozens or hundreds of findings. Triage by severity. Fix CRITICAL findings first, then HIGH. Use suppression rules for findings that are false positives or accepted risks. The goal is a managed queue, not zero findings overnight.

Security Hub
You can't fix what you can't see. This is your security in one view.
Security

Once you enable GuardDuty, Inspector, Macie, and other security services across a multi-account AWS Organisation, findings exist in isolation across dozens of separate dashboards. Security Hub is the aggregation layer that pulls all of those findings into a single pane, normalises them into a common format, and scores your environment against recognised security standards.

What Security Hub provides

Findings aggregationGuardDuty alerts, Inspector CVEs, Macie data exposure findings, IAM Access Analyzer results — all in one place, searchable and filterable.
Security standards scoringContinuous automated checks against CIS AWS Foundations Benchmark, AWS Foundational Security Best Practices, and PCI DSS. You get a score and a prioritised list of controls you're failing.
Cross-account visibilityWith a delegated admin account set up, Security Hub shows you findings across every member account in your organisation from a single console.

How to set it up

01Enable Security Hub from the management account and designate a delegated administrator — typically a dedicated security account. This aggregates findings from all member accounts automatically.
02Enable the AWS Foundational Security Best Practices standard as a baseline. It covers over 200 automated checks across IAM, S3, EC2, RDS, and more. Enable CIS AWS Foundations Benchmark if you have compliance requirements.
03Use Security Hub Insights — saved filters — to create views for your most critical finding types. For example: all CRITICAL findings in prod accounts, all unresolved findings older than 30 days, all failed CIS controls.
04Route Security Hub findings to EventBridge and create automated response playbooks for common finding types using Lambda — for example, automatically revoke a publicly exposed S3 bucket ACL when Security Hub detects one.

CloudTrail
Every action. Every account. Every user. All logged.
Security

CloudTrail records every API call made in your AWS accounts — every CreateInstance, every DeleteBucket, every AssumeRole. It captures who made the call, from which IP address, at what time, with what parameters, and whether it succeeded or was denied. It is the foundational audit trail that every other security service depends on.

What you need to know about the default behaviour

CloudTrail's Event History in the console gives you 90 days of management events for free, but this is not a Trail. It cannot be searched programmatically, cannot be exported, and vanishes after 90 days with no recovery option. For any real use, you need to create an org-wide Trail.

How to set it up properly

01Create an org-wide Trail from the management account. This automatically captures events from all current and future member accounts. Go to CloudTrail → Create Trail → Enable for all accounts in my organisation.
02Point the Trail at an S3 bucket in a dedicated logging account, not the management account. Use a bucket policy that prevents even the management account from deleting logs. This ensures that a compromised management account cannot erase the audit trail.
03Enable Log File Validation. This creates a hash-based digest of every log file, allowing you to detect if any log file has been tampered with or deleted after the fact.
04Enable CloudWatch Logs integration so Trail events are streamed to CloudWatch in near real-time. This enables you to create metric filters and alarms — for example, alert on any root account login, any failed auth attempt exceeding a threshold, or any SCP modification.
05Ship CloudTrail logs to your SIEM or log aggregation tool — Datadog, Splunk, OpenSearch, or similar. Set S3 retention to at least 1 year. Incident investigations and compliance audits regularly require looking further back than 90 days, and having that data indexed and searchable is the difference between a two-hour investigation and a two-week one.
⚠️ Data events cost money. Management events (API calls to AWS services) are free for one Trail. Data events (S3 object-level activity, Lambda invocations) are charged per 100K events. Enable data events selectively — for your most sensitive S3 buckets and critical Lambda functions — rather than org-wide, or your CloudTrail bill will surprise you.

Service Control Policies (SCPs)
The second decision that makes or breaks your compliance.
AWS

SCPs are policies attached to your AWS Organisation, OUs (Organisational Units), or individual member accounts that define the maximum permissions anyone in that account can ever have. They sit above IAM. A permission granted by IAM is still blocked if the SCP doesn't allow it. Even an account's root user is constrained by SCPs when accessed through the organisation.

How they differ from IAM policies

IAM policies say what a specific user, role, or service is allowed to do. SCPs say what is possible to do at all within an account or OU. You use SCPs to draw a hard boundary around the entire account — IAM then manages permissions within that boundary. A developer in the dev account can't accidentally create a resource in a restricted region even if their IAM policy allows it, because the SCP blocks it org-wide.

SCPs to implement from day one

→Require tags on resource creation. Deny the creation of EC2 instances, RDS databases, ECS services, Load Balancers, and S3 buckets unless they include mandatory tags like Environment, Team, and Project. Without this, cost attribution becomes guesswork within weeks of onboarding a second team.
→Enforce EBS volume encryption. Deny the creation of any unencrypted EBS volume. This is one line in an SCP and it prevents a common compliance gap that auditors look for in every SOC 2 and ISO 27001 review.
→Restrict API activity to approved regions. If your operations run in us-east-1, write an SCP that denies all API actions in every other region. This prevents accidental resource provisioning outside your monitored perimeter and dramatically reduces your attack surface. Exemptions can be added for global services like IAM and CloudFront.
→Prevent disabling security services. Write an SCP that denies guardduty:DeleteDetector, cloudtrail:StopLogging, securityhub:DisableSecurityHub, and similar actions. This ensures that even an account administrator cannot disable your security monitoring.
→Deny leaving the organisation. An SCP that denies organizations:LeaveOrganization prevents a member account from detaching itself from your org — which would immediately remove all SCP-based controls from that account.
🔴 Test SCPs in non-prod OUs first. A poorly written SCP can silently block legitimate operations without an obvious error message. Users will just see Access Denied and have no idea why. Apply new SCPs to a test OU, verify nothing breaks, then roll out to prod accounts. Rollback is fast but the blast radius of a bad SCP is everything in that account.
